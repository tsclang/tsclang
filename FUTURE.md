## C Interop: автогенерация .d.tsc из заголовочников

**Идея**: отдельный инструмент `tsclang bind` (или `tsc-bindgen`), который
парсит `.h` файл и генерирует `.d.tsc` с объявлениями.

Пример:
```
tsclang bind sqlite3.h --out sqlite3.d.tsc
```

Аналоги: `bindgen` (Rust), `zig translate-c`.

**Почему откладываем**: ручные `.d.tsc` дают полный контроль над ownership
семантикой (кто владелец, где borrow, какой деструктор). Автогенератор
потребует либо эвристик, либо аннотаций поверх `.h`. Это отдельный
полноценный инструмент. Текущий подход (ручная запись) исключает промахи.

**Когда актуально**: после стабилизации `.d.tsc` формата (фаза 6+).

## Сделать поддержку python-like синтаксиса

стандартный вид:

```
// вариант 1
if (условие) {
    инструкция1 строка1
    инструкция1 строка2
} else {
    инструкции2
}

// вариант 2
if (условие)
    инструкция1
else
    инструкция2

// вариант 3
if (условие) инструкция1
else инструкция2
```

я предлагаю с поддержкой форматирования через табы или пробелы

```
if (условие)
    инструкция1 строка1
    инструкция1 строка2
else
    инструкции2
```

Правила:

1. делаем препроцессор как в кофескрипт
2. если встретил явную {, отключай логику отступов до тех пор, пока не встретишь закрывающую }
3. делать автоматическую вставку точек с запятой `;` в конце строк
4. поддерживать мультистрочные строки через обратные кавычки \` или тройные """, чтобы внутри них препроцессор игнорировал отступы и точки с запятой.

```
if (A)
    if (B)
        step1()
    else
        step2()
const x = 10
```

Шаги препроцессора:

1. if (A) -> добавляет { на следующей строке.
2. if (B) -> добавляет { на следующей строке.
3. step1() -> обычная строка.
4. Видит else (отступ меньше, чем у step1, но слово else) -> не закрывает скобку для if (B).
5. step2() -> обычная строка.
6. Видит const (отступ меньше, чем у else) -> вот теперь вставляет } для if (B) и } для if (A).

Итог для Си:

```
if (A) {
    if (B) {
        step1();
    } else {
        step2();
    }
}
const x = 10;
```

Вот нюансы, которые тебе нужно учесть, чтобы авто-вставка `;` не сломала код на Си:

### 1. Где НЕЛЬЗЯ ставить `;`

В Си точка с запятой после некоторых строк вызовет ошибку или предупреждение. Препроцессор должен пропускать:

* Строки, заканчивающиеся на { или } (если пользователь их использовал).
* Строки с заголовками конструкций: if (...), else, while (...), for (...), switch (...) и другие.
* Строки с определениями функций (перед началом блока).

### 2. Проблема разрыва строки (Line Wrapping)

Если пользователь захочет перенести длинное выражение:

```
const x = 10 +
    20
```

Твой препроцессор может ошибочно вставить `;` после `10 +`.

Решение: Проверяй последний символ строки. Если это оператор (`+`, `-`, `*`, `/`, `,`, `||`, `&&`), то `;` на этой строке не ставится.

### 3. Порядок работы (Алгоритм)

Самый надежный порядок для твоего препроцессора будет таким:

    1. Считаем отступ текущей строки.
    2. Сравниваем со стеком:
        * Больше? Вставляем `{`.
        * Меньше? Вставляем `}` (столько раз, сколько уровней закрылось).
    3. Анализируем содержимое строки:
        * Если это не заголовок (типа `if`) и не открытая скобка, и строка не заканчивается на оператор — добавляем `;` в конец.

Пример трансформации:

```
// Твой код
if (a > 0)
    const x = 5
    const y = 10
```

```
// На выходе для парсера
if (a > 0) {
    const x = 5;
    const y = 10;
}
```

### 4. "Взгляд назад" (Стиль JavaScript/Swift)

Когда препроцессор анализирует строку, он смотрит на её первый символ.

Если строка начинается с оператора (+, -, *, /, &&, ||), значит, это продолжение предыдущей строки.

Действие: Препроцессор не ставит `;` на предыдущей строке.

Как это видит препроцессор:

Строка 1: `const a = 1` (пока не ставим `;`, ждем следующую строку).

Строка 2: Видим `+ 2`. Опа, это оператор! Значит, склеиваем со строкой 1.

Итого:

Если строка заканчивается на оператор (`+`, `,`, `[` и т.д.) — `;` не ставится.

Если следующая строка начинается с оператора (`+`, `\`, `[` и т.д.) — `;` на предыдущей строке не ставится.

Это сделает язык очень похожим на Swift или Go, где точки с запятой есть в итоговом коде, но программист их не пишет.

### 5. В JS есть правило: если после `return`, `throw`, `break` или `continue` идет перенос строки, движок считает, что инструкция закончена, и лепит туда `;`.

Мы выбираем путь Python/Swift: Если после `return` на следующей строке идет отступ (Indent), значит, это продолжение выражения. Но в Си `return` — это ключевое слово, за которым должно следовать выражение на той же строке или в явных скобках.

Если пользователь напишет:

```
return
    value
```

Мы должны добавить скобки по тем же правилам, что и для {}:

```
return
(
    value
)
```

### Где это НЕ должно работать (Исключения):

Чтобы не «сломать» парсинг Си-кода, препроцессор должен игнорировать:

1. Директивы компилятора: Строки, начинающиеся на `#` (например, `#include`), не должны получать `;` в конце.
2. Пустые строки: Просто пропускаем.
3. Комментарии: Если строка — это только комментарий `//` или `/*...*/`, ничего не вставляем.

### Где это должно работать:

Поддерживаются операторы:

- if / else if / else
- for
- do / while
- switch / case
- function

Структуры данных:

- enum
- type, interface

Классы и содержимое классов

- конструктор
- методы
- свойства

Вот пара практических советов

1. Токены-фантомы: В лексере лучше всего генерировать виртуальные токены OPEN_BRACE и CLOSE_BRACE там, где препроцессор видит изменение отступов. Тогда основной парсер даже не будет знать, что в исходнике не было скобок — он будет работать с ними как в обычном TS.

2. Отладка (Source Maps): Поскольку ты добавляешь строки (скобки) и меняешь структуру, пользователю будет сложно дебажить Си-код. Подумай о том, чтобы твой компилятор вставлял в Си-код директивы #line, чтобы ошибки GCC/Clang указывали на правильные строки в твоем исходнике.

3. Контекстный while: Как мы заметили с do-while, твой лексер должен уметь различать «начальный while» и «завершающий while». Проще всего это сделать, если лексер будет хранить флаг inside_do_block.
