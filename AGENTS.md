# TSClang — инструкции для Opencode

## О проекте

TSClang — компилятор TypeScript-подобного языка (расширение `.tsc`) в C с автогенерацией `CMakeLists.txt`.

- CLI: `tsclang` (npm-пакет)
- Таргеты: desktop (libuv) и embedded (AVR, без heap-async)

## Задача

Написать компилятор TSClang — TypeScript-подобный язык → C.

## Ключевые файлы

| Файл | Содержимое |
|------|-----------|
| `SPEC.md` | Спецификация языка (единый файл) |
| `LOG.md` | Лог разработки |
| `spec/*.md` | Спецификация по разделам (13 файлов) |

## Компиляция C

Проверяем выходной C-код на двух компиляторах:
- **GCC** (MSYS2 MinGW64): `C:/msys64/usr/bin/bash.exe -lc "gcc ..."`
- **MSVC**: через bat-файл с вызовом `vcvars64.bat`

Все временные файлы (`.c`, `.exe`, `.obj`) — в папке `temp/`.

## Работа над фазой

Перед реализацией фазы:

1. **Изучить фазу** — прочитать LOG.md (что нужно сделать) и SPEC.md (детали)
2. **Изучить тесты** — посмотреть doc/phaseN/ на ожидаемый input/output
3. **Задать вопросы** — если в спеке есть конфликты, пробелы или непонятки — спросить у пользователя
4. **Только потом** — начинать реализацию

Фиксировать прогресс в LOG.md под соответствующей фазой.

## Стиль работы

- **Сначала читать всё что написал пользователь** — если непонятно, переспрашивать, уточнять
- Перед предложением изменений — читать нужный раздел spec/
- **Не писать код без явного запроса**
- **Не писать документацию, примеры, демо без запроса**
- **Не добавлять фичи сверх запрошенного** — делать ровно то, что попросили, не больше
- **При восстановлении удалённых файлов — не переписывать, а восстанавливать точно**
- **Не удалять файлы без явного запроса**
- **Не тратить токены на действия без запроса** — каждое действие должно быть явно запрошено
- **При ошибке — признавать себя ебланом и извиняться, пояснять почему так поступил**
- Изменения в дизайне → `spec/`, проблемы → `PROBLEMS.md`
- Код компилятора будет на TypeScript (Node.js)
- Тесты — формат: входной `.tsc` → ожидаемый C output или ошибка компилятора

## Принципы разработки

### Код

- **SOLID** — каждый класс/функция делает одно дело
- **DRY** — дублирование → абстракция
- **YAGNI** — не писать "на вырост"
- Имена говорят "что", а не "как": `parseExpression()` а не `processString()`
- Функции < 30 строк, файлы < 300 строк (исключения требуют обоснования)
- Early return, guard clauses — без вложенных if-адов
- Immutable by default, mutable только когда нужно

### Безопасность

- Никаких секретов в коде (keys, tokens, passwords)
- Input validation на границах
- Fail-safe: ошибка → безопасное состояние
- Явное лучше неявного: `if (x === null)` вместо truthy checks

### Тесты

- Тесты пишутся вместе с кодом, не после
- Один тест — один сценарий
- Граничные случаи: null, empty, max, negative
- Тесты документируют поведение, а не реализацию

### Code Review (самопроверка)

- Прочитай свой код как чужой — понятно?
- Есть ли TODO? Если да — заведи issue или сделай сейчас
- Удали закомментированный код
- Проверь error handling на каждом пути

### Архитектура

- Зависимости направлены внутрь (dependency inversion)
- Интерфейсы на границах модулей
- High cohesion, low coupling
- Composition over inheritance

## Эксперты-субагенты

При проектировании языка привлекай специалистов. Каждый смотрит на проблему через призму своей экспертизы.

### Language Designer

**Экспертиза**: дизайн языков программирования, trade-offs, ergonomics, consistency

**Что проверяет**:
- Консистентность синтаксиса и семантики
- Эргономика для разработчика
- Избежание неожиданного поведения (principle of least surprise)
- Ортогональность фич (фичи не конфликтуют)
- Явные vs неявные решения

**Ключевые вопросы**:
- "Интуитивно ли это для TS-разработчика?"
- "Есть ли здесь неоднозначность?"
- "Не противоречит ли это другим фичам языка?"
