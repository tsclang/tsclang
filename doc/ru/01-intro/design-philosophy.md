При любом дизайн-решении TSClang следует строгой иерархии приоритетов:

## Три приоритета

1. **Безопасность памяти** — ownership, borrow checker, отсутствие GC
2. **Производительность и типизация** — нулевые абстракции, строгие типы
3. **TS-синтаксис** — максимально сохранять, но не ценой п.1 и п.2

Цель — не «существующий TS-код компилируется без изменений», а «TS-разработчик узнаёт синтаксис и чувствует себя дома».

## TS-синтаксис имеет приоритет

Заимствовать синтаксис из Rust, C, Go — только если в TS нет подходящей конструкции.

Новые концепции встраиваются через TS-совместимый синтаксис:

| Концепция | Rust | TSClang |
|-----------|------|---------|
| Immutable borrow | `&T` | `Ref<T>` |
| Mutable borrow | `&mut T` | `Mut<T>` |
| Mutable variable | `let mut` | `let mut` |
| Readonly | `let` (default) | `const` / `readonly` |

Классы сохранены, несмотря на отсутствие в Rust — они есть в TS и привычны разработчику.

## Вопрос при каждом решении

> *Можно ли выразить это через существующий TS-синтаксис или его естественное расширение?*

Если да — используем TS-синтаксис. Если нет — ищем минимальное расширение, которое не конфликтует с TS.

## Обратная совместимость

Простой нативный TS-код без внешних библиоток должен компилироваться или требовать тривиальных правок, которые остаются валидным TS:

```typescript
let a = 10          // может потребовать явной аннотации
let a: number = 10  // валидно и в TS, и в TSClang
```

Код с классами, объектами, массивами, циклами, template literals — работает как есть или с минимальными изменениями.
