Моё мнение по проекту **TSClang** — это не просто «пет-проект», а **чрезвычайно амбициозная и технически глубокая попытка создать production-ready язык программирования**. Ниже разбор по ключевым аспектам.

---

### Что это такое
Компилятор с TypeScript-синтаксиса (`.tsc`) в читаемый C, ориентированный на кросс-платформенную разработку: от серверов и десктопа до embedded (AVR, ARM) и ретро-консолей (NES, Sega Genesis, PS1, PS2, DOS, ZX Spectrum). **961 тест**, 20 фаз (0–19), все проходящие.

---

Все 961 тест — полноценные. Из них 7 с пустым expected.out — это корректно (stderr-only, if(false), exit(0) и т.д.). 3 теста-заглушки (generic-partial-param, staticmap-create, cmake-avr) были починены — теперь они проверяют реальный codegen/runtime.

Примечание: фазы 15 (линтер) и 16 (реестр) реализованы, но не имеют собственных тест-кейсов в `test/cases/` — их функционал покрывается CLI-тестами из других фаз.

---

### Сильные стороны (где проект впечатляет)

1. **Продуманная спецификация и дисциплина**  
   `SPEC.md`, `LOG.md`, `CLAUDE.md` — это не формальности, а рабочие инструменты. Фазовая модель (0–19), чёткое разделение на блоки, требование «не добавлять фичи сверх запрошенного» говорит о зрелом подходе к управлению сложностью.

2. **Серьёзная семантика языка**  
   - Ownership / borrow checker (`T`, `Ref<T>`, `Mut<T>`, `Shared<T>` ARC) — прямой заимствование из Rust, но адаптированный под TS-разработчика.  
   - Async/await через state machines в C (без рантайма на embedded).  
   - Generics через монорфизацию, fat pointers для интерфейсов, vtables, extension methods.  
   - Error handling (`throws`, `try/catch`, `?`, `!`) через Result-struct в C, без `setjmp/longjmp`.

3. **Мультиплатформенность не на словах**  
   Не просто «мы скомпилируем в C». Реальные toolchain-файлы (`cmake/toolchain-*.cmake`), платформенные `runtime_*.h`, profile checker с запретом float/heap/async для конкретных таргетов. Это редко встречается даже в зрелых языках.

4. **Реальная стандартная библиотека**  
   `std/fs`, `std/net`, `std/ws`, `std/io` имеют не только codegen, но и реальные C-реализации (POSIX/Win32 сокеты, WebSocket RFC 6455, файловая система). Для компилятора, написанного с нуля — это очень высокая планка.

5. **Тестирование**  
   961 тест с тремя типами: `[F]` (сравнение C-output), `[R]` (компиляция + запуск), `[E]` (ошибки компилятора). Покрытие borrow checker, async state machines, декораторов, ретро-платформ, реального I/O (POSIX/Win32). Это enterprise-уровень тестовой культуры.

6. **Инструментарий**  
   LSP (hover, completion, goto definition), инкрементальная компиляция с SHA-256 кэшем, source maps, declaration emitter, AST-оптимизатор, линтер с автофиксом, package manager с semver — всё это реализовано и протестировано.

### Слабые стороны и риски (где стоит волноваться)

1. **Архитектура codegen: отсутствие IR**  
   В `SPEC.md` фаза 14 упоминает IR, но в `LOG.md` честно сказано: «Без SSA IR — всё через C-backend. Оптимизации делегируем gcc». Фаза 14 помечена как завершённая (`[x]`), но IR в ней не реализован — вместо этого добавлены name mangling, incremental compilation, optimization levels.  
   Это значит, что весь компилятор — это **прямая трансляция AST → C-строки**. При текущем масштабе (~693 KB JS в `src/compiler/`) это работает, но при добавлении:
   - серьёзных оптимизаций,
   - сложного dead code elimination,
   - продвинутого анализа lifetime,
   - inline assembly / SIMD,
   
   …отсутствие промежуточного представления станет архитектурным bottleneck. Рефакторинг будет болезненным.

2. **Крупные файлы кодогенерации**  
   Все основные монолиты разбиты на модули:
   ```
   calls/        9 модулей  (макс. 38 KB)  — из 128 KB
   stmt/         5 модулей  (макс. 72 KB)  — из 128 KB
   top-level/    6 модулей  (макс. 25 KB)  — из 118 KB
   async/        5 модулей  (макс. 17 KB)  — из 55 KB
   expr/         4 модуля   (макс. 27 KB)  — из 50 KB
   types/        3 модуля   (макс. 33 KB)  — из 48 KB
   misc/         4 модуля   (макс. 15 KB)  — из 33 KB
   ```
   Максимальный модуль — `stmt/vardecl.js` (72 KB), крупнейший монолитный метод — `inferType` (33 KB).

3. **Парсер 60KB — самописный, без использования готовых инфраструктур**  
   Это и плюс (полный контроль), и минус. Нет использования TypeScript Compiler API, tree-sitter или ANTLR. При добавлении сложных фич (например, полноценных generic constraints с dependent types) поддержка парсера станет затратной.

4. **Некоторые «фасадные» фичи**  
   `std/hal`, `std/avr` — desktop stubs, реальная реализация «через platform profile». Platform packages (`@nes/platform`, `@sony/ps1` и т.д.) отложены. Это нормально для этапа разработки, но пока язык не может действительно прошиваться в железо без ручной работы.

**Вердикт:** Это один из самых впечатляющих solo/малых-командных проектов в области языков программирования, который я видел. Автор не просто «написал транспилер», а построил **полноценный системный язык с borrow checker-ом, async runtime, мультиплатформенным кодгеном и пакетным менеджером**.

Единственное, что может убить проект в долгосрочной перспективе — это **архитектурный технический долг в codegen**: прямая эмиссия C из AST без IR. Все основные монолитные файлы разбиты на модули (макс. 72 KB, большинство ≤38 KB), но `types.js` (48 KB) и `misc.js` (33 KB) остаются крупными. Для выхода на следующий уровень (production embedded, оптимизации, сложный C-interop) потребуется либо рефакторинг с выделением линейного IR, либо миграция бэкенда на что-то вроде LLVM/MLIR/QBE.

Но на текущем этапе (фазы 0–19, 961 тест) — это **феноменальный результат**.
